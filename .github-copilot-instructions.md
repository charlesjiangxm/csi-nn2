### Architecture Relationships

**Data Flow**:
1. **User Application** → calls CSI-NN2 API (include/csinn/csi_nn.h)
2. **NN2 Layer** (source/nn2/*.c) → dispatcher selects backend
3. **Backend Selection** → determined by compile flags (CONFIG_BUILD_* in CMake/Kconfig)
   - Reference backend (source/reference)
   - Architecture-specific (source/c906_opt, c920_opt, etc.)
   - Vector extensions (source/thead_rvv)
   - Matrix extensions (source/thead_matrix)
4. **Backend Registration** → Each backend provides setup.c that registers callbacks
5. **Graph-Level Optimization** (source/graph_ref) → optional graph scheduling layer
6. **LLM Support** (source/llm) → specialized for large language models

**Backend Hierarchy** (from source/nn2/setup.c):
```
shl_init() selectively enables:
├── #ifdef SHL_BUILD_REF → shl_target_init_ref()
├── #ifdef SHL_BUILD_GREF → shl_target_init_gref()
├── #ifdef SHL_BUILD_C906 → shl_target_init_c906()
├── #ifdef SHL_BUILD_C920 → shl_target_init_c920()
├── #ifdef SHL_BUILD_RVV → shl_target_init_rvv()
├── #ifdef SHL_BUILD_RVM → shl_target_init_rvm()
├── #ifdef SHL_BUILD_PNNA → shl_target_init_pnna()
└── ... (other backends)
```

## 2. BUILD CONFIGURATION SYSTEM

### Three-Layer Configuration Approach

#### Layer 1: **Kconfig** - Declarative Feature Selection
- **File**: `/Kconfig` (main), `source/*/Kconfig` (backend-specific)
- **Purpose**: Allows interactive `.config` generation via `make menuconfig`
- **Key Selections**:
  - Build target (X86, RISC-V, C-SKY, RISC-V ELF)
  - Platform variant (C906, C908, C920, C920V2, RVV, RVM, E907, etc.)
  - Per-layer feature toggles: `CONFIG_C_REFERENCE_<OP>_<DTYPE>`, `CONFIG_C906_<OP>_<DTYPE>`, etc.
  - Debug/profiling options: `CONFIG_USE_SHL_DEBUG`, `CONFIG_SHL_LAYER_BENCHMARK`, `CONFIG_SHL_TRACE`

#### Layer 2: **CMake** - Build System Integration
- **Files**: `CMakeLists.txt` (root), `source/*/CMakeLists.txt`, `cmake/*.cmake`
- **Process**:
  1. Root CMakeLists.txt loads optional `config.cmake` from build directory or source directory
  2. Conditionally includes target-specific configs (e.g., `cmake/c906_static.cmake`)
  3. Includes backend-specific CMakeLists.txt which conditionally add source files based on Kconfig
  4. Example (from `source/reference/CMakeLists.txt`):
     ```cmake
     if(CONFIG_C_REFERENCE_SOURCE)
         list(APPEND REF_SRCS_MOD source/reference/utils.c)
     endif()
     if(CONFIG_C_REFERENCE_ABS)
         list(APPEND REF_SRCS_MOD source/reference/abs.c)
     endif()
     ```

#### Layer 3: **Makefile** - Convenience Targets
- **File**: `/Makefile`
- **Purpose**: Abstraction over CMake for common build scenarios
- **Key Targets**:
  ```makefile
  make nn2_c906           # Build C906 static library
  make nn2_c906_so        # Build C906 shared library
  make nn2_c920           # Build C920 static library
  make nn2_rvv            # Build generic RVV library
  make nn2_ref_x86        # Build x86 reference library
  make menuconfig         # Interactive Kconfig editor
  ```
- **Implementation Pattern**: Each target creates build directory, runs CMake with specific CONFIG flags, then make

### Compiler Selection (cmake/rules.cmake)

**Automatic based on target**:
- RISC-V Linux: `riscv64-unknown-linux-gnu-{gcc,g++}`
- RISC-V ELF: `riscv64-unknown-elf-{gcc,g++}`
- C-SKY ELF: `csky-abiv2-elf-{gcc,g++}`
- Override: Set `CONFIG_USE_COMPILER_PATH=ON` in config.cmake

### Optimization Flags by Target

**Example from CMakeLists.txt**:
- **X86 Ref**: `-mavx -mfma -fopenmp -DSHL_AVX_OPT`
- **C906**: `-march=rv64gcv0p7_zfh_xtheadc -mabi=lp64d -DSHL_BUILD_C906`
- **C920**: `-march=rv64gcv0p7_zfh_xtheadc -mabi=lp64d -DSHL_BUILD_C920 -fopenmp`
- **RVV generic**: `-march=rv64gcv_zfh_xtheadc_xtheadvdot -mabi=lp64d -DSHL_BUILD_RVV`
- **RVM**: `-march=rv64gcv_zfh_xtheadc_xtheadvdot_xtheadmatrix -mabi=lp64d -DSHL_BUILD_RVM`

### Static vs. Shared Library
- **Control**: `CONFIG_SHL_BUILD_STATIC` flag
- **CMake logic** (cmake/target_build.cmake):
  ```cmake
  if(CONFIG_SHL_BUILD_STATIC)
      add_library(${SHL_LIB_TARGET} STATIC ...)
  else()
      add_library(${SHL_LIB_TARGET} SHARED ...)
  endif()
  ```
- **Makefile pattern**: Static by default; append `_so` target for shared lib

## 3. SOURCE CODE ORGANIZATION CONVENTIONS

### NN2 Core Layer (source/nn2/)
- **Role**: Single-point API dispatch layer
- **Pattern**: Each operation has `<op>.c` that:
  1. Validates inputs (dimensions, data types, quantization parameters)
  2. Calls `params->cb.<op>` callback registered by active backend
  3. Provides fallback or default behavior if backend not available
- **Key file**: `setup.c` - registers all backend initialization functions
- **Quantization Support**: Symmetric and asymmetric per-tensor or per-channel quantization via `struct csinn_quant_info`

### Backend Implementation (source/<backend>/)

**Shared Pattern**:
- `setup.c` (~350-2000 lines) - registers operation callbacks in `struct csinn_target` via dispatch table
- `utils.c` - backend-specific utility functions (memory allocation, quantization conversion, etc.)
- Operations organized by:
  - **Data type** (fp32/, fp16/, int8/) for optimized backends
  - **Operation** (one .c/.S file per operation)
- Assembly files (.S) for performance-critical paths (e.g., convolution, matrix multiply)

**Example Setup Pattern** (source/c906_opt/setup.c):
```c
target->ops[CSINN_OP_ABS].init = shl_c906_abs_init;
target->ops[CSINN_OP_ADD].init = shl_c906_add_init;
target->ops[CSINN_OP_CONV2D].init = shl_c906_conv2d_init;
// ... one init function per operation
```

### Reference Backend (source/reference/)
- **Purpose**: Portable C implementation for validation and fallback
- **Organization**: One operation per file (add.c, conv2d.c, etc.), no dtype-specific variants
- **Used for**: Accuracy validation, cross-backend comparison, CPU inference

### RISC-V Vector Backend (source/thead_rvv/)
- **Purpose**: Common RVV optimization layer shared by C906, C908, C920, etc.
- **Organization by dtype**:
  - fp32/ - full-precision operations
  - fp16/ - half-precision operations
  - int8/, int32/, int4/ - quantized operations
- **Vectorization**: Uses RISC-V V extension intrinsics and assembly
- **Reusability**: C920 = RVV + C920-specific optimizations; C906 = RVV + C906-specific optimizations

### Graph Reference Backend (source/graph_ref/)
- **Purpose**: Graph-level optimization (operator fusion, memory planning, scheduling)
- **TVM Integration**: Optionally integrates with TVM for code generation (controlled by `CONFIG_GRAPH_REFERENCE_TVMGEN`)
- **Key file**: `setup.c` - graph-level operation dispatch and fusion decisions

## 4. TESTING AND VALIDATION

### Test Harness Categories

#### A. **Layer Validation Tests** (tests/validation_layer/)
- **Type**: Operation-level correctness tests
- **Makefiles**: `Makefile.ref`, `Makefile.c906`, `Makefile.c920`, `Makefile.rvv`, etc.
- **Pattern**:
  - Compile against specific backend library (e.g., `../../c906_static_build/libshl_c906.a`)
  - Each operation has test file (add.c, conv2d.c, etc.) that:
    1. Generates random test data using Python (tests/python_ref/<op>.py)
    2. Calls SHL function
    3. Compares output against Python reference
  - Test data files: `valid_datas/<op>_<dtype>_<variant>_*.dat`
- **Execution**: Via Makefile (e.g., `make test_c906`)

#### B. **Graph Validation Tests** (tests/validation_graph/)
- **Type**: End-to-end model inference tests
- **Backends Tested**: pnna, anole, c906, th1520, tvmgen, hlight
- **Makefiles**: `Makefile.pnna`, `Makefile.c906`, etc.

#### C. **Python Reference Framework** (tests/python_ref/)
- **~300 Python operation files** for generating test data
- **Pattern**: Each file exports function like `add_f32()` or `add_i8()` that:
  - Generates random parameters (shape, quantization ranges, etc.)
  - Runs NumPy/reference computation
  - Outputs binary test data (.dat files)
- **Invoked by**: Makefile test targets as `python3 python_ref/<op>.py <variant> > test_data`

#### D. **Pytest Automation** (tests/autotest/)
- **Main File**: `interface_test.py`
- **Configuration**: `conftest.py` - pytest hooks to accept `--board`, `--dtype`, `--accuracy` parameters
- **Pattern**:
  - Pytest discovers test functions
  - Each test generates data via Python reference
  - Compiles and runs C validation binary
  - Compares against ONNX reference (via tests/onnx_ref/)
- **Example invocation**:
  ```bash
  pytest --board=c906 --dtype=8 --accuracy=0.99 interface_test.py
  ```

#### E. **Unit Tests** (tests/unit_test/)
- **Makefile.rvv** - unit tests for RVV-specific functions (intrinsic wrappers, etc.)

#### F. **Profiling** (tests/profiler/)
- Performance benchmarking of layer kernels
- Makefile: `Makefile`

### CSINN_API Levels (tests/validation/Makefile.* define CSINN_API)
- **API=0**: Reference backend (CSINN_REF)
- **API=3**: Optimized backend (e.g., C906 for Makefile.c906)

## 5. KEY CODING CONVENTIONS

### Data Structures (include/csinn/csinn_data_structure.h)
```c
struct csinn_tensor {
    void *data;
    int dtype;               // CSINN_DTYPE_FLOAT32, INT8, UINT8, etc.
    struct csinn_quant_info *qinfo;  // Quantization parameters
    struct csinn_session *sess;
    // ... dimension info
};

struct csinn_session {
    struct csinn_model model;
    struct csinn_tensor **input, **output;
    // ... execution context
};

struct csinn_params_base {
    struct csinn_callback *cb;   // Backend callback functions
    struct csinn_session *sess;
};
```

### Operation Function Signature Pattern
```c
// Initialization (from NN2 layer)
int csinn_<op>_init(struct csinn_tensor *input,
                     struct csinn_tensor *output,
                     struct csinn_<op>_params *params);

// Backend implementation (called via callback)
int shl_<backend>_<op>_init(...);  // Captures input/output metadata
int shl_<backend>_<op>(...);        // Execution kernel (called multiple times)
```

### Quantization Handling
- Per-tensor: single `qinfo` with scale and zero_point
- Per-channel: array of `qinfo`, one per output channel
- Conversion between formats handled in utils.c of each backend

### Assembly Code (.S files)
- Used for performance-critical operations (convolution, matmul)
- RISC-V syntax (e.g., c906, c920, rvv)
- Linked into static/shared library
- Registered as fallback if C version slower

### Debugging Flags
- `CONFIG_USE_SHL_DEBUG` → `-DSHL_DEBUG` → enables debug logging in shl_debug.h
- `CONFIG_SHL_TRACE` → `-DSHL_TRACE` → generates execution trace
- `CONFIG_SHL_LAYER_BENCHMARK` → `-DSHL_LAYER_BENCHMARK` → logs layer execution time

## 7. EXAMPLE AND DOCUMENTATION

### Example Programs (example/)
- **Pattern**: Standalone C programs that use SHL API
- **Typical Structure**:
  ```c
  #include "csi_nn.h"
  
  // 1. Create session and tensors
  struct csinn_session *sess = csinn_alloc_session(...);
  struct csinn_tensor *input = csinn_alloc_tensor(...);
  
  // 2. Initialize operation
  csinn_<op>_init(input, output, params);
  
  // 3. Set data and run
  input->data = data;
  csinn_<op>(input, output, params);
  ```
- **Compilation** (example/Makefile):
  ```makefile
  riscv64-unknown-linux-gnu-gcc prog.c -I../include -o prog.elf \
      ../install_nn2/c906/lib/libshl_c906.a -lm -static
  ```
- **Documentation**: API docs at https://csi-nn2.opensource.alibaba.com/

## 8. IMPORTANT NOTES FOR CONTRIBUTORS

### Adding a New Operation
1. **Reference Implementation**: Add `source/reference/<op>.c`
2. **NN2 Dispatcher**: Add entry to `source/nn2/<op>.c` and `source/nn2/setup.c`
3. **Kconfig**: Add `CONFIG_C_REFERENCE_<OP>` toggle to `source/reference/Kconfig`
4. **Backend Optimizations** (optional):
   - `source/c906_opt/<op>.c` or fp32/int8 subdirectories
   - Add to `source/c906_opt/Kconfig` and CMakeLists.txt
5. **Tests**:
   - Python reference: `tests/python_ref/<op>.py` with test data generator
   - C validation: `tests/validation_layer/<op>.c` linking against backend library
   - Add to Makefile.c906, Makefile.ref, etc.
6. **Documentation**: Update include files and API docs

### Multi-Backend Build Strategy
- **CMake** orchestrates based on Kconfig selections
- **Each backend is independent**: Can be compiled separately or together
- **NN2 dispatch layer** ensures seamless fallback if backend unavailable
- **Graph layer** optional for cross-backend optimization

### Build Output Locations
- **Default**: `<backend>_build/` and `<backend>_build_so/` directories
- **Installation**: `../install_nn2/<backend>/` (via `make install_nn2`)
- **Libraries**: Named `libshl_<backend>.a` (static) or `libshl_<backend>.so` (shared)

### Performance Profiling
- Use `CONFIG_SHL_LAYER_BENCHMARK` to measure per-layer execution time
- Use tests/profiler/ for detailed performance analysis
- Assembly implementations prioritized for convolution, matmul, reduce operations
